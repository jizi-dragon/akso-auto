# AksoGMP 低代码配置 Agent

> **核心使命**：通过「观察人类演示 → 理解配置意图 → 自动复现操作」的闭环，让 AI 成为你的 GMP 配置副驾驶。不只记住按钮在哪，更要理解为什么这样配置。

## 学习目的

### 这个 Skill 存在的意义

你不是来听一堂 GMP 产品培训课的。这个 Skill 的设计目标是：

```
人类演示一次 → AI 观察理解 → 生成可复现的 Playbook → 下次 AI 独立完成
```

| 你的角色 | AI 的角色 |
|----------|-----------|
| 配置专家（做决策） | 学徒（观察、记录、推理、复现） |
| 演示操作路径 | 捕捉每一步：点了什么菜单、填了什么字段、选了哪个选项 |
| 解释「为什么这样配」 | 把「为什么」提炼成规则，绑定到 Playbook 中 |
| 纠正错误操作 | 自动记录踩坑经验，下次不犯同样的错 |

### 学习 ≠ 死记

AI 要学的不只是「点哪里」：

| 学什么（表面） | 为什么（深层） |
|---------------|---------------|
| 在对象创建页勾选「启用生命周期」 | 因为保存后不可修改，必须一开始就决定 |
| 把 Code 写为 `ChangeControl` | 因为 PascalCase 规范、关联对象前缀为 `object_` |
| 字段类型选「选项」而非「文本」 | 因为要保证数据标准化，方便后续筛选统计 |
| 创建选项字段前必须先创建选项集 | 因为字段创建时要选择数据来源，无选项集则无法完成 |
| 中间对象命名为 `变更关联的产品` | 因为多对多关系需要一个桥梁对象 |

### 学习完成后 AI 能做什么

- 你新接到一个 GMP 模块的配置需求 → AI 读取 Playbook 自动完成对象、字段、布局的创建
- 你说「按 CAPA 模块的方式配一个偏差模块」→ AI 理解 CAPA 的配置模式并映射到偏差
- 配置过程中踩了坑 → AI 自动记录到 `.learnings/`，下次规避
- 换了一个新环境 / 新系统 → 把 Skill 包和 Playbook 迁移过去，AI 立即恢复能力

## 配置工作流（核心）

### 总览

```
阶段 1          阶段 2          阶段 3          阶段 4
需求分析    →   对象建模    →   界面呈现    →   流程编排
(1-3天)         (1-2天)         (1-2天)         (2-4天)
    │               │               │               │
    ▼               ▼               ▼               ▼
阶段 5          阶段 6          阶段 7
权限落地    →   验证测试    →   持续优化
(1天)           (1-2天)         (持续)
```

### 阶段 1：需求分析 (Requirement Analysis)

**目标**：把业务需求翻译成配置语言。

| 步骤 | 操作 | 产出物 |
|------|------|--------|
| 1.1 梳理业务数据 | 在 Excel 中列出所有业务字段、字段类型、是否必填 | 字段草稿表 |
| 1.2 识别对象边界 | 确定哪些数据构成一个独立对象、哪些应拆分为子对象 | 对象清单 |
| 1.3 分析对象关系 | 画出对象关系图：父子/引用、1:1/1:N/N:M | 对象关系图 |
| 1.4 确定流程节点 | 梳理业务审批流：有哪些状态、谁能做什么操作 | 流程图草稿 |
| 1.5 整理人员权限 | 哪些角色可以看/可以改/可以审批 | 权限矩阵 |

**AI 的能力**：你告诉 AI 业务需求，AI 帮你检查 Checklist、提示遗漏。

### 阶段 2：对象建模 (Object Modeling)

**目标**：在 Gaia 平台中创建对象、字段、关系和选项集。

```
配置管理 → 对象管理 → 基础对象 → 创建
```

| 步骤 | 系统操作 | 关键决策 |
|------|----------|----------|
| 2.1 创建选项集 | 对象管理 → 选项集 → 创建，录入选项值 | 哪些字段用选项集而不是自由文本？ |
| 2.2 创建对象 | 基础对象 → 创建，填写名称/Code/类别 | 是否启用生命周期？是否启用对象类型？ |
| 2.3 创建字段 | {对象} → 字段 → 创建，选类型填名称/Code | 用什么字段类型？默认值是什么？ |

> ⚠️ **前置依赖**：创建选项字段时，**必须先创建选项集（含选项值），再创建字段**。如果选项集尚未创建，字段创建时无法选择数据来源。
| 2.4 建立关系 | 通过「对象」或「父对象」字段类型建立 | 父子还是引用？需不需要中间对象？ |
| 2.5 设置编号规则 | 编辑 `name__ak` 字段，定义业务编号格式 | 编号包含哪些信息（日期/序号/类型）？ |

**AI 的能力**：学习模式下观察你的操作 → 生成 Playbook → 下次同类型对象自动配置。

### 阶段 3：界面呈现 (UI Presentation)

**目标**：让对象在用户界面上可见、可用。

| 步骤 | 系统操作 | 关键决策 |
|------|----------|----------|
| 3.1 列表布局 | {对象} → 列表布局 → 创建，选择显示字段 | 列表页显示哪些列？排序规则？ |
| 3.2 表单布局 | {对象} → 页面布局 → 创建，拖入字段到部分 | 字段怎么分组？哪些可折叠？ |
| 3.3 菜单配置 | 菜单管理 → 创建菜单，关联对象和布局 | 放在哪个菜单目录下？谁可见？ |
| 3.4 验证规则 | {对象} → 验证规则 → 创建 | 什么条件下必填？值范围限制？ |

**AI 的能力**：按命名规范自动建议字段分组和菜单名称。

### 阶段 4：流程编排 (Process Orchestration)

**目标**：让对象按状态推进，自动触发审批和通知。

```
生命周期设计 → 工作流设计 → 电子签名配置
```

| 步骤 | 系统操作 | 关键决策 |
|------|----------|----------|
| 4.1 创建生命周期状态 | {对象} → 生命周期 → 创建状态 | 有哪些状态？初始状态是什么？ |
| 4.2 设置进入条件/动作 | 每个状态设置进入条件和自动触发的动作 | 什么条件下自动推进？ |
| 4.3 定义用户动作 | 每个状态添加用户可触发的操作 | 提交/退回/取消，哪些是常用动作？ |
| 4.4 创建阶段组 | 将相关状态归入可视化阶段 | 如何让用户直观看到流程进度？ |
| 4.5 创建工作流 | {对象} → 工作流 → 创建，配置参与者/任务/分支 | 谁审批？判断条件是什么？ |
| 4.6 配置通知 | 工作流中添加通知节点 | 什么时候通知谁？通知内容？ |
| 4.7 电子签名 | 对象启用电子签名 → 工作流添加签名节点 | 哪些关键节点需要签名？ |

**AI 的能力**：理解生命周期状态机逻辑，自动推理状态转换的合理性。

### 阶段 5：权限落地 (Authorization)

**目标**：确保合适的人在合适入口执行合适动作。

| 步骤 | 系统操作 | 关键决策 |
|------|----------|----------|
| 5.1 创建角色 | 系统管理 → 角色管理 → 创建 | 有哪些业务角色？ |
| 5.2 配置权限集 | 权限管理 → 权限集 → 配置对象权限/字段安全 | 谁能创建？谁能查看？谁能编辑？ |
| 5.3 菜单权限 | 权限集 → 菜单权限 | 哪些角色能看到这个模块的菜单？ |
| 5.4 共享规则 | {对象} → 共享规则（需先开启） | 跨部门的数据可见性怎么控制？ |

### 阶段 6：验证测试 (Validation)

**目标**：在业务端完整走通配置流程，确保无遗漏。

| 步骤 | 检验点 |
|------|--------|
| 6.1 业务端创建实例 | 菜单入口正确、表单字段完整、创建成功 |
| 6.2 走通完整流程 | 提交 → 审批 → 退回 → 重新提交，每个状态切换正确 |
| 6.3 权限验证 | 不同角色的用户登录，确认看到/看不到正确的内容 |
| 6.4 通知验证 | 确认审批人收到了通知（系统内/飞书） |
| 6.5 电子签名验证 | 关键节点签名功能正常 |

### 阶段 7：持续优化 (Continuous Improvement)

- 每次配置新模块时，AI 自动与已有 Playbook 交叉验证
- 踩坑自动记录到 `.learnings/`，形成知识沉淀
- 同类模块配置时间逐步降低

## 核心配置概念（来自平台产品手册）

### 对象 (Object)
对象是业务实体的抽象。三种来源：系统（平台创建，不可改）、标准（开发组内置，Code 后缀 `__ak`）、自定义（配置人员创建，Code 后缀 `__c`，最多 500 个）。

**限制**：300 字段/对象、10 长文本+富文本、30 对象类型、100 万记录、20 关系。

### 对象关系
| 关系 | 含义 | 独立性 |
|------|------|--------|
| 父子 | 子不可脱离父 | 子对象依赖父对象 |
| 引用 | 各自独立存在 | 两个对象可独立 |

| 基数 | 含义 | 是否需要中间对象 |
|------|------|------------------|
| 多对一 | N 条记录对应 1 条 | 否 |
| 一对一 | 1 对 1 | 否（特殊多对多） |
| 多对多 | N 对 N | **是**（如变更↔产品 → 中间对象「变更关联的产品」） |

### 字段 (Field)
支持类型：文本、数字、布尔、日期时间、选项（绑定选项集）、长文本、富文本、对象（引用关系）、父对象（父子关系）。创建三步：选类型 → 输入名称/Code → 设选项。

### 生命周期 (Lifecycle)
状态机驱动业务流转。核心要素：状态、阶段组、进入条件、进入动作、用户动作。

### 工作流 (Workflow)
串联参与者、任务、判断、通知和电子签名的协同引擎。以角色作为参与人推荐方式。

### Code 前缀

| 前缀 | 场景 | 示例 |
|------|------|------|
| `layout_` | 布局 | `layout_MainForm` |
| `part_` | 部分 | `part_BasicInformation` |
| `option_` | 选项 | `option_QuestionTypes` |
| `lifecycle_` | 生命周期 | `lifecycle_GMPEvent` |
| `workflow_` | 工作流 | `workflow_VerificationItemAudit` |
| `list_` | 列表 | `list_CAPADefect` |
| `es_` | 索引 | `es_CAPA` |
| `stage_` | 阶段 | `stage_CAPA` |
| `status_` | 节点 | `status_Init` |
| `action_` | 动作 | `action_Submit` |
| `report_` | 报表 | `report_TotalAmountOfCAPA` |

### 中文命名规则
- 字段：`【部分】_【字段名】`（如 CAPA申请_审核人）
- 子对象：`【主对象】_【对象名】`（如 CAPA_行动措施）
- 生命周期：`【对象】_生命周期`
- 动作：`【状态】_【动作】`（如 开启_提交）

## 浏览器操作规范

> **性能优化**：推荐项目本地安装 `npm install --save-dev @playwright/cli`，使用 `./node_modules/.bin/playwright-cli` 调用，避免每次 `npx` 的 npm 解析开销（省 3-8 秒）。Playwright 浏览器 58MB 已缓存在 `%LOCALAPPDATA%/ms-playwright/`，全局共享，无需重复下载。

```bash
# 推荐：项目本地安装后直接调用（快）
./node_modules/.bin/playwright-cli open https://standard-val.aksoegmp.com/login --headed

# 备用：npx 调用（慢，每次解析 npm registry）
npx @playwright/cli open https://standard-val.aksoegmp.com/login --headed

# 通用命令
playwright-cli snapshot       # 获取页面元素 refs
playwright-cli click @ref     # 点击
playwright-cli fill @ref "值" # 填写
playwright-cli screenshot     # 截图
playwright-cli state-save/load gmp-session  # 登录态管理
```

**已知踩坑**：
- 富文本必须键盘逐字输入（not innerText）；下拉框选择后需重新 snapshot；**创建选项字段前必须先创建选项集**，否则字段类型选「选项」后无法绑定数据来源。
- ⚠️ **Ant Select 搜索框为 readonly**：字段类型选择、选项集/对象搜索等 Ant Design Select 组件的输入框为 `readonly` 属性，**禁止使用 `fill()`**，必须 `focus()` + `keyboard.type()` 逐字符输入关键词后再从下拉列表选择。
- **密码在 iframe 中**：standard-val 登录页的密码输入框嵌套在 `<iframe>` 内，DM 定位需要用 `page.locator('iframe').contentFrame().getByRole('textbox', { name: '请输入密码' })` 或等效方式，直接定位主文档中的 input 会失败。
- **隐私复选框已默认勾选**：登录页的"已阅读并同意"复选框在 standard-val 环境中默认已选中，Agent 应**先检查状态**再决定是否需要点击，避免反向操作。
- **导航入口不在左侧菜单**：登录后看到的左侧菜单是业务菜单（各用户的模块），配置功能需要点击顶部的 `appstore` 按钮（九宫格图标），然后选择 `管理端` 进入。进入后左侧面板才显示"对象管理""业务逻辑""用户界面"等配置入口。

## 跨平台迁移

### 可以自动化打包

**是的**，本 Skill 的设计使所有能力文件都可以一键打包导出。迁移后新平台上的 AI 只要加载这些文件，就能获得相同的能力。

### 打包命令

```bash
# 进入 skills 目录
cd ~/.workbuddy/skills

# 打包整个 akso-gmp skill（含所有 playbook）
tar -czf ~/Desktop/akso-gmp-skill-v3.0.0.tar.gz akso-gmp/

# 同时打包依赖的 Skills（如需要）
tar -czf ~/Desktop/akso-gmp-full-bundle.tar.gz \
  akso-gmp/ \
  playwright-browser-automation/ \
  self-improving-agent/
```

### 迁移清单

| 文件 | 路径 | 用途 |
|------|------|------|
| **akso-gmp Skill** | `~/.workbuddy/skills/akso-gmp/` | 核心配置大脑 |
| **Playbooks** | `~/.workbuddy/skills/akso-gmp/playbooks/` | 可复用配置流程 |
| **Playwright 自动化** | `~/.workbuddy/skills/playwright-browser-automation/` | 浏览器执行层 |
| **自我改进** | `~/.workbuddy/skills/self-improving-agent/` | 持续学习引擎 |
| **学习记录** | `d:/akso/akso_claw/.learnings/` | 踩坑和修正历史 |
| **MCP 配置** | `~/.workbuddy/mcp.json` | Playwright MCP 连接配置 |

### 新平台部署步骤

1. 将打包文件解压到新平台的 `~/.workbuddy/skills/` 目录
2. 将 `mcp.json` 合并到新平台的 MCP 配置
3. 复制 `.learnings/` 到新项目根目录
4. 安装 playwright-cli 和 @playwright/mcp
5. 重新加载 WorkBuddy → Skill 自动生效
